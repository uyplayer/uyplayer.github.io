Unix and Linux operating systems provide the fork () system call to create a process. The fork () function is called by the parent process, and the child process is created. The child process is a copy of the parent process, but the child process has its own PID. . The fork () function is very special. It returns twice. The parent process can get the PID of the child process through the return value of the fork () function. The return value in the child process is always 0. Python’s os module provides a fork () function. Because there is no fork () call in Windows system, to achieve cross-platform multi-process programming, you can use the Process class of the multiprocessing module to create sub-processes, and this module also provides higher-level encapsulation, such as the process pool for batch starting processes Pool), Queues and Pipes used for inter-process communication.

Multithreading in Python

In earlier versions of Python, the thread module (now named _thread) was introduced to implement multi-threaded programming. However, this module is too low-level and many functions are not provided. Therefore, we recommend using the threading module for current multi-threaded development. Modules provide better object-oriented encapsulation for multithreaded programming. Because multiple threads can share the memory space of the process, it is relatively simple to implement communication between multiple threads. The most direct way you can think of is to set a global variable. Multiple threads can share this global variable. But when multiple threads share the same variable (we usually call it “resource”), it is likely to produce uncontrollable results and cause the program to fail or even crash. If a resource is used by multiple threads competing, then we usually call it a “critical resource”, and access to the “critical resource” needs to be protected, otherwise the resource will be in a “chaotic” state. The following example demonstrates the scenario where 100 threads transfer to the same bank account (transfer of 1 yuan). In this example, the bank account is a critical resource. Without protection, we are likely to get errors. result.

Running the above program, the results were astounding, 100 threads each transferred 1 yuan to the account, and the result was actually much less than 100 yuan. The reason this happens is because we do not protect the “critical resource” of the bank account. When multiple threads deposit money into the account at the same time, the new_balance = self._balance + money line is executed together. Multiple threads The account balances obtained are all 0 in the initial state, so all +1 operations are performed on 0, and thus wrong results are obtained. In this case, “lock” can come in handy. We can protect “critical resources” through “locks”. Only the threads that acquire the “locks” can access the “critical resources”, and other threads that do not obtain the “locks” can only be blocked until the threads that obtain the “locks” are released. With the “lock”, other threads have the opportunity to acquire the “lock” and then access the “critical resources” that are protected. It is a pity that the multi-threading of Python can not take advantage of the multi-core characteristics of the CPU. This can be confirmed by starting a few threads that execute the infinite loop. The reason for this is because the Python interpreter has a “global interpreter lock” (GIL) thing. Any thread must first acquire a GIL lock before executing it, and then every 100 bytes of code are executed, the interpreter automatically releases the GIL lock. Letting other threads have the opportunity to execute is a problem left over from history, but even so, as in our previous example, the use of multithreading is still positive in improving execution efficiency and improving the user experience.
